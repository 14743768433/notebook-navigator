# 文件夹内笔记层级

## 背景

Notebook Navigator 当前的文件夹视图主要是平铺列表：一个文件夹下的笔记按某种规则排序显示。

这个模型适合浏览普通文件，但不适合表达“笔记与笔记之间的上下级关系”。用户希望在同一个文件夹里，把笔记组织成类似 Lattics 的树状结构：

```text
进程管理
  程序如何变成线程/进程？
  CPU如何切换和调度线程 进程 等等
  线程进程之间如何进行通信
    进程之间如何通信
    线程之间的通信
      共享内存实现的原理
        进程之间的同步
同步
  线程的同步
  临界区和锁相比优势
```

这里每一个条目仍然是一个真实笔记，但笔记之间可以存在父子层级。

## 目标

在文件夹视图中支持笔记级别的层级结构。

目标体验：

1. 同一个文件夹里的笔记可以显示为树状结构。
2. 每个节点都是一篇真实笔记，而不是普通标题或虚拟分组。
3. 笔记可以有子笔记。
4. 用户可以展开或折叠有子节点的笔记。
5. 用户可以通过拖动调整同层级内的顺序。
6. 用户可以通过拖动把一篇笔记移动为另一篇笔记的子节点。
7. 拖动过程中必须能清楚区分“换顺序”和“变成子节点”这两种意图。

## 设计参考与调研记录

调研日期：2026-06-30

### Lattics

Lattics 是当前需求的主要体验参考，而不是源码参考。

需要借鉴的是它在项目/卡片/笔记管理中呈现出的层级化组织体验：

1. 笔记或卡片可以被组织成有上下级关系的结构。
2. 用户在浏览时能直接看到上下文层级，而不是只能看到平铺列表。
3. 层级结构服务于写作、学习和知识组织，而不是单纯模拟文件系统。

当前未找到 Lattics 的官方开源源码仓库，因此不能把它作为实现参考或代码参考。

参考链接：

- https://lattics.com/en-US/

### Trilium / TriliumNext

Trilium 是一个开源的层级笔记应用，官方文档明确把 note tree 作为核心交互。

可借鉴点：

1. 笔记可以在树中拖拽重排。
2. 支持通过键盘上下移动调整同级顺序。
3. 支持通过键盘左右移动改变层级。
4. 展开/折叠树节点是默认导航能力。

这说明“笔记本身作为树节点”是成熟交互，不只是文件夹树的变体。

参考链接：

- https://docs.triliumnotes.org/user-guide/concepts/ui/note-tree
- https://github.com/TriliumNext/trilium

### Obsidian Bases Link Structure

`Bases_Link_Structure` 是一个 Obsidian 插件，用 `up` 属性表达父节点，并把笔记显示为可拖拽的层级树。

可借鉴点：

1. 使用 frontmatter/link 属性表达父子关系。
2. 拖拽树节点后自动更新父子链接。
3. 支持一个笔记出现在多个父节点下。
4. 支持多选移动。
5. 使用 Obsidian 原生链接格式，降低和现有 vault 的冲突。

它和本需求非常接近，但它是独立的结构视图；本需求希望把类似能力整合进 Notebook Navigator 的文件夹列表体验中。

参考链接：

- https://github.com/Fertion/Bases_Link_Structure

### Obsidian Hierarchical TOC

`obsidian-hierarchical-toc` 使用 `parent` frontmatter 字段显示当前笔记的子节点树。

可借鉴点：

1. `parent` 字段是表达父子关系的直接方案。
2. 层级关系可以不依赖物理文件夹结构。
3. 一个笔记可以有多个父节点，从严格树扩展成更灵活的知识图谱。

它更偏“当前笔记的子节点目录”，不是完整的文件夹列表改造。

参考链接：

- https://github.com/andresgongora/obsidian-hierarchical-toc

### Dendron

Dendron 是开源的层级优先 Markdown 笔记系统，用点号命名表达层级，例如 `recipes.italian.desserts.md`。

可借鉴点：

1. 层级优先的查找、创建和导航模型。
2. 子笔记在父笔记上下文中可见。
3. 层级结构可以和本地 Markdown 文件共存。

但 Dendron 的层级主要来自文件命名规则；本需求暂不希望通过重命名文件来表达父子关系。

参考链接：

- https://wiki.dendron.so/notes/4RFQMNA91cRW12VJl_PFd/

### Logseq / SiYuan

Logseq 和 SiYuan 都是开源或开放源码生态中重要的层级化知识管理产品。

可借鉴点：

1. Logseq 证明了大纲式层级编辑在知识管理中是核心交互。
2. SiYuan 证明了块级结构、文档树、双链和本地优先可以结合。

它们更偏 block/document 级系统，不是直接对应 Obsidian 文件夹列表，但可作为层级编辑体验和大纲交互的背景参考。

参考链接：

- https://github.com/logseq/logseq
- https://github.com/siyuan-note/siyuan

## 非目标

这份 spec 暂不要求：

1. 把真实文件移动到子文件夹中。
2. 改变 vault 的物理目录结构。
3. 把层级关系同步到 Obsidian 原生文件树。
4. 一次性支持跨文件夹拖动成为子节点。
5. 一次性支持标签视图、属性视图、搜索结果中的层级编辑。
6. 用普通 Markdown 标题替代笔记节点。

## 数据模型要求

层级关系应该表达在笔记元数据中，且不依赖物理文件夹层级。

需要进一步设计的数据字段可能包括：

1. `sort_index`：继续表达同一父节点下的顺序。
2. `parent` 或类似字段：表达当前笔记的父笔记。
3. 可能需要稳定的笔记标识符，避免只用标题导致重命名后关系断裂。

需要注意：

1. 如果使用文件路径作为父节点引用，重命名或移动文件时必须更新子节点引用。
2. 如果使用唯一 ID，需要考虑已有笔记如何初始化 ID。
3. 层级关系不能形成循环，例如 A 是 B 的父节点，同时 B 又是 A 的父节点。

## 交互要求

### 树状显示

文件夹视图可以显示为树状列表：

1. 有子节点的笔记左侧显示展开/折叠控件。
2. 子节点相对父节点缩进。
3. 当前选中笔记仍然按现有方式高亮。
4. 折叠父节点后，其子节点暂时隐藏。
5. 展开/折叠状态应在当前会话内保留，是否持久化需要后续确认。

### 同级换顺序

当用户拖动笔记到另一个笔记的上方或下方时，系统应理解为同级排序。

视觉反馈应明确表示：

1. 插入到目标节点之前。
2. 插入到目标节点之后。
3. 插入位置位于哪个层级。

保存结果：

1. 被拖动笔记的父节点保持不变。
2. 同一父节点下相关笔记的 `sort_index` 更新。

补充定稿：第一期允许“跨层级同级插入”。插入线所在的缩进层级就是放手后的目标层级；如果把子节点拖到根级插入线，放手后它会变回根级笔记。如果插入线位于某个父节点的子级缩进，放手后它会成为该父节点下的同级节点。

### 移动为子节点

当用户拖动笔记到另一个笔记“内部”或“缩进区域”时，系统应理解为把它变成目标笔记的子节点。

视觉反馈应明确表示：

1. 当前操作不是普通换顺序。
2. 被拖动笔记将成为哪个目标笔记的子节点。
3. 放下后会出现在目标父节点下的哪个位置。

保存结果：

1. 被拖动笔记的父节点字段更新为目标笔记。
2. 被拖动笔记在新父节点下获得新的 `sort_index`。
3. 原父节点下剩余子节点顺序保持稳定并按需重新编号。

### 区分两种拖动意图

这是本需求的关键。

拖动反馈必须让用户在放手之前知道系统会执行哪一种操作：

1. 如果是换顺序，显示横向插入线。
2. 如果是变成子节点，突出目标父节点或显示缩进后的预览位置。
3. 鼠标水平位置、目标条目垂直区域、拖动停留时间或修饰键可以作为判定依据，但最终规则必须稳定、可学习。
4. 不应出现用户以为是在换顺序，结果变成子节点的情况。
5. 不应出现用户以为是在变成子节点，结果只是换顺序的情况。

## 边界情况

需要处理：

1. 不能把笔记拖成自己的子节点。
2. 不能把笔记拖到自己的后代节点下面，避免形成循环。
3. 折叠节点参与拖动时，需要明确是插入到折叠节点旁边，还是放入折叠节点内部。
4. 多选笔记拖动是否支持，暂时待定。
5. 搜索、过滤、隐藏文件、非 Markdown 文件存在时，层级编辑规则需要避免产生不可见父子关系误操作。
6. 如果父笔记被删除、移动或重命名，子笔记关系需要有恢复或清理策略。

## 验收标准

1. 同一个文件夹内的笔记可以显示为树状层级。
2. 每个树节点都对应一篇真实 Markdown 笔记。
3. 有子节点的笔记可以展开和折叠。
4. 用户可以拖动笔记调整同级顺序。
5. 用户可以拖动笔记成为另一篇笔记的子节点。
6. 拖动过程中，换顺序和变成子节点有明显不同的视觉反馈。
7. 拖动完成后，层级关系和同级顺序会被持久化。
8. 切换文件夹再回来，树状层级仍然正确。
9. 重启 Obsidian 后，树状层级仍然正确。
10. 系统会阻止形成循环层级。
11. 普通点击打开文件、键盘导航、搜索、筛选等基础列表行为不因树状层级退化。

## 已确认的决定（2026-06-30 讨论）

原“需要进一步确认的问题”已逐条定稿，结论如下。

1. **层级关系存储：集中 JSON（方案 A），NN 专有，不写进笔记 frontmatter。**
   - 数据形态：**版本化 JSON envelope** `{ version, parents: { 子: 父 }, updatedAt }`，`parents` **严格单值单父**（未来多父时把 `parents` 的值升级为数组，靠 `version` 做迁移）。`updatedAt` 仅供人工调试同步冲突，不参与自动合并。
   - **`sort_index` 继续维护**：父子关系走 JSON，同级顺序走 frontmatter 的 `sort_index`，**两套都保留**。树 = 用 JSON 组装层级，每个父节点下的兄弟节点用 `sort_index` 排序。
   - **存储位置：单独文件**，路径 `.obsidian/plugins/notebook-navigator/hierarchy.json`，由 NN 自己读写（不走 `saveData` 的 data.json）。
   - **同步**：该文件可被 Obsidian 官方 Sync 带走，但需用户在 `Settings → Sync` 勾选社区插件相关的 vault 配置同步项；且“插件目录内额外文件是否同步”并非官方文档明确保证，属已知风险。冲突处理为整文件 last-write-wins（无行级合并）；单人多设备场景冲突概率极低。
   - 选择 JSON 而非 frontmatter `up` 字段（如 `Bases_Link_Structure` 插件做法）的理由：符合 NN “元数据集中存”的既有风格，不污染笔记；缺点是层级关系为 NN 专有，换插件/用 Bases 看不到这棵树——已接受。
2. **严格单父**：一篇笔记只能有一个父节点。多父（树退化为图）暂不支持，留作后续（届时将 `父路径` 改为 `父路径[]`）。
3. **仅当前文件夹内部允许建立父子关系**（与“非目标”一致）。
4. **第一期不显示跨物理文件夹的树**：树 = 当前文件夹内的笔记按 JSON 关系组装。跨文件夹层级是独立的后续功能。
5. **“拖成子节点”判定：以指针水平缩进位置为主**——拖到目标行左侧对齐区 = 同级排序（画横向插入线）；向右越过缩进阈值 = 成为上一个节点的子节点（高亮目标父节点 + 缩进预览）。判定规则需在实现前定死，且全程有放手前预览，避免误操作。
   - **同级插入线允许跨层级放置**：插入线在哪个缩进层级，放手后就成为该层级的同级节点；这也是把子节点拖回根级的第一期主路径。
6. **折叠节点被拖入时，新子节点放到子列表末尾**（追加，`sort_index` 取 `lastRank + STEP`，无需压缩）。
7. **提供“提升/降低层级”键盘快捷键（Alt+←/→），但延后**：第一期先做拖动；该能力是现有 Alt+↑/↓ 同级移动的自然延伸。
8. **批量操作分阶段**：第一期拖动排序支持多选（复用默认拖动排序）；**批量“变为某节点子节点”延后**，先把单个变子节点跑通。
9. **Pinned notes 不参与层级**：第一期 pinned notes 始终平铺置顶，不参与树结构展示，也不参与层级拖动；树只作用于普通文件区。这一刀切避免了三个歧义——pinned 父节点的子节点是否跟显、子节点被 pin 了算不算子节点、pin 区参与层级拖动是否会改父子关系。

### 展开/折叠状态

- **持久化，跨重启保留**（复用 NN 导航树已有的展开状态持久化机制 `ExpansionContext` 模式）。原 spec“交互要求 / 树状显示”中“展开/折叠状态是否持久化需后续确认”一项据此定稿。

## 实现要点（详见 [实现计划](implementation-plan.md)）

- 本功能**建立在默认拖动排序之上**，复用 `src/utils/manualSort.ts` 数据层与 `@dnd-kit` 拖动框架；顺序：拖动排序 → 层级数据层 → 层级交互层。
- 父子关系由独立的 `HierarchyService`（版本化 `hierarchy.json` + 防环 + **变更订阅**）维护；路径在重命名/删除时经 `registerWorkspaceEvents` 与 metadata/recent **并列**同步；父删除时子节点回退根级。
- `sort_index` **不重新分作用域**，仅在“同一父的兄弟集合”内相对使用，复用现有分数排名/插入/压缩逻辑零改动。
- 树在 `listItems` 构建链路按 `parent + sort_index` 组装为带 `depth` 的扁平序列；**有关系才自动成树**，无关系即扁平可拖列表。
- 两意图拖动用指针**水平位移**判定（横向插入线 vs 缩进预览），放手前必须有预览；防环在落点即时校验。
- 折叠状态复用 `ExpansionContext` 的 localStorage 持久化（已跨重启）。
- `hierarchy.json` 位于 `.obsidian/` 内，不出现在 vault 文件列表，无需在 NN 列表里过滤。

文件/函数级步骤、风险与测试策略以 [实现计划](implementation-plan.md) 为准。
