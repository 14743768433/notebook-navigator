# 默认拖动排序

## 背景

Notebook Navigator 当前在文件夹列表中支持手动排序，但交互更像一个临时编辑模式：

1. 用户默认打开文件夹时看到普通文件列表。
2. 用户需要点击顶部的排序/拖动按钮进入“手动排序: sort_index”模式。
3. 进入模式后，列表顶部会显示说明文字和“完成”按钮。
4. 用户拖动条目调整顺序后，需要点击“完成”退出该模式。

这个流程能完成排序，但会让“拖动排序”显得像一个单独任务，而不是文件夹列表的自然能力。

## 目标

把文件夹列表中的手动拖动排序变成默认、自然、低打扰的交互。

目标体验：

1. 打开文件夹后，列表仍然保持普通浏览状态。
2. 用户可以直接拖动 Markdown 笔记调整顺序。
3. 拖动完成后自动写入对应笔记的 `sort_index`。
4. 不再长期显示“手动排序: sort_index”的说明区块。
5. 不需要用户点击“完成”来结束拖动排序。
6. 文件夹列表默认使用紧凑模式，而不是标准模式。

换句话说，目标流程从：

```text
点击进入手动排序模式 -> 阅读提示 -> 拖动 -> 点击完成
```

变成：

```text
打开文件夹 -> 直接拖动排序 -> 自动保存
```

## 非目标

这份 spec 暂不要求：

1. 改变 `sort_index` 的数据格式。
2. 改变非 Markdown 文件无法参与手动排序的规则。
3. 改变文件夹、标签、属性等其他导航视图的排序逻辑。
4. 重新设计整个顶部工具栏。
5. 添加新的同步或冲突解决机制。

## 交互要求

### 默认状态

文件夹列表默认显示为普通浏览状态，不出现大段排序说明。

顶部工具栏可以保留排序相关入口，但不应该要求用户先进入一个显式的“排序模式”才能拖动。

### 默认列表密度

文件夹列表的默认显示密度应改为紧凑模式，而不是标准模式。

当前项目已有 `defaultListMode` 设置项，可选值包括 `standard` 和 `compact`。目标是让未单独设置外观覆盖的文件夹默认使用
`compact`。

要求：

1. 新安装或未显式选择列表模式的用户，默认看到紧凑列表。
2. 紧凑模式下优先显示笔记标题，减少日期、描述、预览文本等信息造成的纵向占用。
3. 已经对某个文件夹、标签或属性节点单独设置为标准模式的外观覆盖，不应该被默认值强行改掉。
4. 用户仍然可以通过现有外观入口切换回标准模式。
5. 默认紧凑模式应服务于“直接拖动排序”的体验，让列表更像可快速扫描、可直接操作的笔记目录。

### 拖动行为

当当前文件夹列表满足手动排序条件时：

1. 用户可以直接拖动列表里的 Markdown 文件。
2. 拖动时应有清晰的视觉反馈，表示当前条目正在移动。
3. 拖动结束后，列表顺序立即稳定在用户放置的位置。
4. 新顺序应自动保存到文件 frontmatter 的 `sort_index` 字段。

### 不可排序项

如果当前列表中存在不能参与 `sort_index` 排序的项目，例如非 Markdown 文件：

1. 这些项目仍按现有规则处理。
2. 不应该通过常驻说明文字占用列表顶部空间。
3. 如果需要提醒，应采用低打扰方式，例如临时 toast、tooltip、状态提示或只在用户尝试拖动不可排序项时提示。

### 保存反馈

拖动完成后的保存反馈应尽量轻量。

可以考虑：

1. 成功时不显示任何常驻提示。
2. 保存失败时显示错误通知，并尽量恢复到保存前顺序或给出明确状态。
3. 保存过程中避免阻塞用户继续浏览，除非存在数据一致性风险。

## 验收标准

1. 打开文件夹后，列表顶部不再显示“手动排序: sort_index”的常驻说明区块。
2. 打开文件夹后，不需要点击“手动排序”按钮即可拖动 Markdown 文件重新排序。
3. 没有单独外观覆盖的文件夹默认以紧凑模式显示，而不是标准模式。
4. 用户仍然可以手动切换回标准模式，并且显式设置过的标准模式不会被默认紧凑模式覆盖。
5. 拖动 Markdown 文件后，对应文件的 `sort_index` 会被更新并持久化。
6. 切换到其他文件夹再回来，排序结果仍然保留。
7. 重启 Obsidian 后，排序结果仍然保留。
8. 非 Markdown 文件的现有排序限制不被破坏。
9. 保存失败时用户能看到错误反馈，而不是静默失败。
10. 普通点击打开文件、键盘导航、搜索、筛选等基础列表行为不因默认拖动排序而退化。

## 已确认的决定（2026-06-30 讨论）

原“需要进一步确认的问题”已逐条定稿，结论如下。

1. **触发范围：仅当文件夹排序方式已是 `sort_index` 时启用直接拖动。** 不做全局开关。
   - 实用场景：大多数文件夹不需要排序；只有少数“项目类”文件夹需要在里面工作、清晰看到文件关系。
   - **每文件夹持久化**：用户对某个文件夹手动切到 `sort_index`
     后，NN 自动记住该文件夹的排序方式，下次进来仍是可拖状态，无需重复开启。该持久化能力**已存在**（见下文架构说明），本 spec 不新造。
2. **顶部“手动排序”按钮：保留，但改变语义。**
   它从“进入/退出临时编辑模式”的开关，变为**排序方式入口**——用户通过它把文件夹切到
   `sort_index`（即开启可拖）。一旦切换，列表常驻可拖。显式编辑模式概念、顶部说明面板、“完成”按钮一并移除。
3. **拖动后不自动切换排序方式。** 因为按第 1 条，能拖的前提就是已经是 `sort_index`，不存在“在非 `sort_index`
   列表里拖”的情况，没有自动切换的触发点。
4. **第一期只做文件夹视图。** 搜索结果、标签视图、属性视图暂不支持直接拖动排序（保留未来扩展空间）。
5. **保留多选批量拖动排序。** 复用现有多选拖动实现。
6. **默认紧凑模式影响所有视图**（文件夹/标签/属性/搜索）。因为是单人使用，**直接全量把默认值改为
   `compact`，不做老数据迁移**；已有的逐节点外观覆盖仍然保留生效。

## 实现要点（详见 [实现计划](implementation-plan.md)）

- 复用现有手动排序数据层 `src/utils/manualSort.ts`（`sort_index`
  分数排名，格式不变）与“原位重排 + 保存”管道（`ListPane.tsx` 的 `propertyKeyboardReorder`，当前仅绑键盘）。
- 每文件夹排序方式**已持久化**（`folderSortOverrides`，`sortUtils.ts:492`），“开了下次还在”零成本满足。
- **核心决策：退役非虚拟化的 `ManualSortListContent` 与显式编辑模式 `manualSortEditState`，把指针拖动直接并入已虚拟化的
  `ListPaneVirtualContent`**——从根上消除“常驻渲染器卡顿”风险（而不是给旧渲染器补虚拟化）。
- 虚拟列表拖拽采用保守方案：`useDraggable + useDroppable + DragOverlay + 自定义插入线`，**不依赖 dnd-kit sortable
  strategy**。
- 默认 `compact`：`src/settings/defaultSettings.ts:300` 改默认值，不做迁移；逐节点覆盖仍生效。
- 非 Markdown 不可排序项不加常驻文字，必要时一次性 toast；保存失败弹 warning toast，成功无提示。

文件/函数级步骤、风险与测试策略以 [实现计划](implementation-plan.md) 为准。
